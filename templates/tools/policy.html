1{% if language == 'pl' %}Wprowadzenie{% else %}Introduction{% endif %}
{% if language == 'pl' %}

Niniejsza Polityka Bezpieczeństwa Informacji (dalej: Polityka) określa zasady ochrony informacji {% if company_name %}w organizacji {{ company_name }}{% else %}w organizacji{% endif %} zgodnie z wymogami Dyrektywy (EU) 2022/2555 (NIS2), implementowanej do prawa polskiego ustawą o krajowym systemie cyberbezpieczeństwa.

Polityka obowiązuje wszystkich pracowników, współpracowników oraz podwykonawców, którzy mają dostęp do systemów informatycznych i danych organizacji. Niniejszy dokument stanowi punkt startowy — przed wdrożeniem wymaga przeglądu prawnego i dostosowania do specyfiki organizacji.

{% else %}

This Information Security Policy sets out the rules for protecting information {% if company_name %}at {{ company_name }}{% else %}within the organization{% endif %} in accordance with Directive (EU) 2022/2555 (NIS2) requirements.

This Policy applies to all employees, contractors, and subcontractors with access to the organisation's IT systems and data. This document is a starting point — legal review and tailoring to the organisation's specifics are required before implementation.

{% endif %}

3{% if language == 'pl' %}Odpowiedzialności{% else %}Responsibilities{% endif %}
{% if language == 'pl' %}
  • Właściciel / Zarząd: Zatwierdza Politykę, zapewnia zasoby, odpowiada za zgodność z NIS2
  • Osoba odpowiedzialna za IT: Wdraża środki techniczne, zarządza dostępami, prowadzi przeglądy
  • Wszyscy pracownicy: Przestrzegają zasad Polityki, zgłaszają incydenty i podejrzane zdarzenia
  • Podwykonawcy IT: Zobowiązani umową do przestrzegania standardów bezpieczeństwa organizacji
{% else %}
  • Owner / Management: Approves this Policy, provides resources, accountable for NIS2 compliance
  • IT responsible person: Implements technical controls, manages access, conducts reviews
  • All employees: Comply with Policy rules, report incidents and suspicious activity
  • IT subcontractors: Contractually bound to follow the organisation's security standards
{% endif %}
4{% if language == 'pl' %}Klasyfikacja aktywów{% else %}Asset Classification{% endif %}
{% if language == 'pl' %}

Wszystkie aktywa informacyjne klasyfikowane są według wrażliwości:

  • Poufne: dane klientów, umowy, dane finansowe, dane osobowe — dostęp ograniczony, szyfrowanie wymagane
  • Wewnętrzne: procedury, korespondencja wewnętrzna — dostęp tylko dla pracowników
  • Publiczne: materiały marketingowe, dane kontaktowe — bez ograniczeń

Inwentarz aktywów jest aktualizowany co roku lub po każdej istotnej zmianie infrastruktury.

{% else %}

All information assets are classified by sensitivity level:

  • Confidential: customer data, contracts, financial data, personal data — restricted access, encryption required
  • Internal: procedures, internal correspondence — employee access only
  • Public: marketing materials, contact information — unrestricted

The asset inventory is updated annually or after any significant infrastructure change.

{% endif %}
5 {% if language == 'pl' %}Kontrola dostępu (MFA i hasła){% else %}Access Control (MFA & Passwords){% endif %}
{% set access_gap = gaps | selectattr('req_id', 'in', ['req_10', 'req_4']) | first | default(None) %} {% if access_gap and access_gap.risk_level in ['critical', 'high'] %}
{% if language == 'pl' %}PILNE: Ocena wykazała lukę w kontroli dostępu ({{ access_gap.risk_level | upper }}). Poniższe zasady wymagają natychmiastowego wdrożenia.{% else %}URGENT: Assessment found an access control gap ({{ access_gap.risk_level | upper }}). Rules below require immediate implementation.{% endif %}
{% endif %} {% if language == 'pl' %}
  • MFA (uwierzytelnianie dwuskładnikowe): obowiązkowe na wszystkich kontach e-mail, VPN, systemach chmurowych i panelach administracyjnych
  • Hasła: minimum 12 znaków, kombinacja liter, cyfr i znaków specjalnych; zmiana co 12 miesięcy lub natychmiast po incydencie
  • Zasada minimalnych uprawnień: każdy pracownik ma dostęp wyłącznie do zasobów niezbędnych do pracy
  • Konta administratorów: oddzielne od kont codziennego użytku; logowanie działań jest wymagane
  • Odejście pracownika: konta dezaktywowane w dniu zakończenia współpracy
{% else %}
  • MFA (multi-factor authentication): mandatory on all email accounts, VPN, cloud systems, and admin panels
  • Passwords: minimum 12 characters, mix of letters, digits, and special characters; changed every 12 months or immediately after an incident
  • Least-privilege principle: each employee has access only to resources needed for their role
  • Admin accounts: separate from daily-use accounts; activity logging required
  • Employee departure: accounts deactivated on the last day of employment
{% endif %}
6{% if language == 'pl' %}Kopie zapasowe{% else %}Backup Policy{% endif %}
{% set backup_gap = gaps | selectattr('req_id', 'equalto', 'req_9') | first | default(None) %} {% if backup_gap and backup_gap.risk_level in ['critical', 'high'] %}
{% if language == 'pl' %}PILNE: Ocena wykazała lukę w obszarze kopii zapasowych ({{ backup_gap.risk_level | upper }}). Zasady poniżej wymagają natychmiastowego wdrożenia i pierwszego testu.{% else %}URGENT: Assessment found a backup gap ({{ backup_gap.risk_level | upper }}). Rules below require immediate implementation and first test.{% endif %}
{% endif %} {% if language == 'pl' %}
  • Częstotliwość: kopia pełna — co tydzień; kopia różnicowa — codziennie dla danych krytycznych
  • Reguła 3-2-1: 3 kopie, na 2 różnych nośnikach, 1 poza siedzibą firmy (chmura lub fizyczna lokalizacja)
  • Szyfrowanie: wszystkie kopie zapasowe muszą być zaszyfrowane
  • Test odtworzenia: przeprowadzany co kwartał; wyniki dokumentowane i przechowywane przez 12 miesięcy
  • Odpowiedzialny: osoba ds. IT lub wyznaczony podwykonawca
{% else %}
  • Frequency: full backup — weekly; differential backup — daily for critical data
  • 3-2-1 rule: 3 copies, on 2 different media, 1 offsite (cloud or physical location)
  • Encryption: all backups must be encrypted
  • Restore test: conducted quarterly; results documented and retained for 12 months
  • Responsible: IT person or designated subcontractor
{% endif %}
7{% if language == 'pl' %}Reagowanie na incydenty{% else %}Incident Response{% endif %}
{% set ir_gap = gaps | selectattr('req_id', 'equalto', 'req_5') | first | default(None) %} {% if ir_gap and ir_gap.risk_level in ['critical', 'high'] %}
{% if language == 'pl' %}PILNE: Ocena wykazała brak procedury reagowania na incydenty ({{ ir_gap.risk_level | upper }}). Szczegółowy plan zawiera dokument „Procedura reagowania na incydenty".{% else %}URGENT: Assessment found no incident response procedure ({{ ir_gap.risk_level | upper }}). See the "Incident Response Plan" document for details.{% endif %}
{% endif %} {% if language == 'pl' %}
  • Definicja incydentu: każde zdarzenie naruszające lub mogące naruszyć bezpieczeństwo systemów (atak ransomware, wyciek danych, nieautoryzowany dostęp, utrata urządzenia)
  • Zgłaszanie: każdy pracownik zgłasza podejrzenie incydentu bezpośrednio do osoby odpowiedzialnej za IT — niezwłocznie, nie później niż w ciągu 4 godzin
  • Obowiązek NIS2: znaczące incydenty muszą być zgłoszone do CSIRT NASK w ciągu 24 godzin od wykrycia (wstępne powiadomienie), finalna ocena — 72 godziny
  • Dokumentacja: każdy incydent dokumentowany w rejestrze incydentów; przechowywany przez minimum 5 lat
  • Szczegółowa procedura: zawarta w odrębnym dokumencie „Procedura reagowania na incydenty"
{% else %}
  • Incident definition: any event breaching or potentially breaching system security (ransomware, data leak, unauthorised access, lost device)
  • Reporting: every employee reports suspected incidents to the IT responsible person immediately, within 4 hours at the latest
  • NIS2 obligation: significant incidents must be reported to the competent CSIRT within 24 hours of detection (preliminary notification), final assessment — 72 hours
  • Documentation: every incident logged in the incident register; retained for a minimum of 5 years
  • Detailed procedure: see the separate "Incident Response Plan" document
{% endif %}
8{% if language == 'pl' %}Przegląd roczny{% else %}Annual Review{% endif %}
{% if language == 'pl' %}

Niniejsza Polityka podlega przeglądowi co najmniej raz w roku lub po każdym znaczącym incydencie bezpieczeństwa, zmianie struktury organizacyjnej lub istotnej zmianie w infrastrukturze IT. Osoba odpowiedzialna za IT przygotowuje raport z przeglądu; zarząd zatwierdza zmiany.

{% else %}

This Policy is reviewed at least once a year, or after any significant security incident, organisational change, or major change to IT infrastructure. The IT responsible person prepares the review report; management approves changes.

{% endif %}
